# Experiment 0006 — grouped device-resident DeepSeek-V4 MoE

Status: **in progress**.

## Hypothesis and contract

The resident DeepSeek-V4 runtime is dominated by a host-synchronous CUDA graph
boundary, not by storage. Keeping each selected expert's W1/W3, clipped SwiGLU,
routed scaling, and W2 intermediate on its assigned GPU—and enqueueing all
active GPUs before observing any result—should materially improve exact decode
throughput without changing placement, routing, precision, or accumulation
order.

- Primary metric: median exact decode steps/s over three interleaved 128-token
  `Hello` repetitions. Success requires at least 5.0 steps/s.
- Correctness: `make check`; native FP4/FP8 operation comparison; frozen
  top-logit and per-layer BF16 traces; identical prompt, route, and greedy token
  IDs; zero decode checkpoint reads.
- Memory: no more than 216 GiB host RSS and the same `0.85` per-device VRAM
  fraction. Device MoE scratch must remain inside the admitted workspace
  reserve.
- Rollback: reject on any semantic, numerical, admission, or zero-NVMe failure,
  or when the median improvement is not material outside observed variance.

The branch is `exp/dsv4-device-moe`, based on observability revision `746e63c`.
The existing host-mediated path remains available in the same binary as the
reference. DSpark remains disabled.

## Evidence and projected effect

A logits-producing forward performs 1,244 CUDA matmuls:

- 297 attention projections;
- 43 router projections;
- 774 routed-expert projections;
- 129 shared-expert projections;
- one vocabulary projection.

Every current projection uploads a host activation, optionally quantizes it,
launches one scalar-style matvec, downloads the output, and synchronizes. The
baseline therefore executed 157,988 matmuls and 161,846 synchronizations over
127 decode forwards, with 2.924 GB activation H2D and 2.374 GB D2H.

The per-device decode kernel totals sum to 49.259 seconds. Because the current
runtime dispatches devices serially, that is 388 ms of kernel service per
forward and an idealized ceiling near 2.58 steps/s before host work.

The frozen smoke route has 2.73 active expert devices per layer on average.
Grouping at that real dependency boundary projects:

- 903 routed/shared projection completions replaced by about 117 per-device
  MoE command completions per forward;
- about 63% fewer total synchronization boundaries;
- about 84% less routed/shared activation H2D and 50% less D2H;
- concurrent service across active GPUs without host worker dispatch.

These are structural projections, not measured results.

## Design

For each layer, the reference router still produces the exact six expert IDs
and coefficients. The runtime acquires W1/W3/W2 as completion-protected cache
leases, groups them by existing device assignment, and submits one command per
active GPU. A command:

1. uploads and E4M3-quantizes the normalized hidden row once;
2. evaluates native FP4 W1 and W3 together with per-32 E8M0 scale reuse;
3. preserves the BF16 boundaries, asymmetric SwiGLU limit 10, and twice-applied
   routed coefficient;
4. E4M3-quantizes the resident intermediate and evaluates native FP4 W2;
5. optionally evaluates the native FP8 shared expert on the layer device;
6. returns separately BF16-rounded routed rows plus the shared row.

The host combines routed rows in original router-rank order, then adds the
shared row at the existing BF16 boundary. Floating atomic accumulation and
reordered cross-device sums are forbidden.

The cache subtracts the admitted 256 MiB workspace reserve from each device's
weight capacity and proves that every device can lease a worst-case six-expert
set after its pinned spine. Move-only leases prevent eviction until every
accepted command has been collected or drained. The backend downloads into
persistent pinned staging and copies to caller memory only after confirmed
completion. Non-finite W1/W3 results are reported as exact failures; they are
never silently clamped into a usable activation.

Observability distinguishes logical projections from grouped commands and
records command launches, transfers, bytes, maximum cumulative per-device
service time, and the measured host wall time across each complete multi-GPU
MoE batch. Cache snapshots expose acquire/release counts, active leases, and
leased bytes. Successful prefill and decode require balanced lease counts and
zero outstanding lease state.

DwarfStar informed the command graph and fixed-top-k fusion pattern. Its
IQ2_XXS, Q2_K, Q4_K, Q8_K activation format, device-wide CUDA barriers, and
single-GPU assumptions are not imported.

## Pre-benchmark validation

- `make check`: 2/2 CTest targets and 55/55 unit cases passed.
- A separate CUDA-disabled build passed the same 55/55 cases.
- Routed FP4 and shared FP8 operation outputs matched the existing three-matmul
  host-SwiGLU sequence bit-for-bit at the BF16 boundary; maximum observed drift
  was exactly 0.0.
- Error fixtures proved that a non-finite E8M0 scale fails explicitly, caller
  output remains untouched, and the persistent workspace is reusable after a
  confirmed drain.
- A warning-enabled `-Werror` C++/CUDA build completed successfully.
- The A/B harness passed both injected-failure aggregation and a synthetic
  successful reference/candidate matrix, including token, route, memory,
  zero-read, lease, command-count, and throughput gates.

## Planned decision sequence

1. Freeze an instrumentation-only reference logit/layer trace.
2. Pass the synthetic CUDA operation oracle and `make check`.
3. Run one full-model correctness/performance screen in a named tmux session.
4. Run the interleaved three-repetition matrix only if the screen and projected
   remaining ceiling justify it; otherwise record the failure and move the next
   bounded branch to a device-resident attention/mHC layer block.
